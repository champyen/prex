# Backtrace System

The Prex+ backtrace system is designed to provide call stack information for both kernel and user-space programs during panics, assertions, and unhandled exceptions. It is built to work reliably across different memory configurations (MMU and NOMMU) and processor architectures.

## 1. System Architecture

The backtrace system operates on a "Proactive Push" model for user-space and direct unwinding for the kernel.

### 1.1 Kernel Backtrace
When a panic or assertion occurs in the kernel, the kernel unwinds its own call stack using the architecture-specific unwinder.

### 1.2 User-Space Backtrace ("Proactive Push")
Due to the complexities of cross-address-space unwinding during a kernel panic, user-space handles its own unwinding:
1. **Self-Unwinding**: During an exception, assertion failure, or user panic, the user-space library unwinds its own stack.
2. **Push to Kernel**: The user-space library calls `backtrace_save()` which invokes the `DBGC_SAVEBT` debug syscall. This pushes the call stack addresses (PC and function entry point) into the kernel's Task Control Block (TCB) for the current task.
3. **Kernel Reporting**: If the kernel subsequently panics (or handles a fatal exception), it retrieves the user stack from the TCB and resolves the function names.

### 1.3 Name Resolution
Function names are resolved directly from memory rather than parsing complex DWARF debugging information. On supported architectures (e.g., ARM), the compiler is configured to embed function name strings immediately preceding the function's entry point (using `-mpoke-function-name`). The kernel's `backtrace_name()` reads this embedded string using the function entry addresses stored in the TCB.

## 2. API Design

The backtrace API is defined in `<sys/backtrace.h>` and is shared between kernel and user-space.

### Unwinding APIs
* `int backtrace_unwind(backtrace_t *buffer, int size)`
  Unwinds the current call stack and populates the `buffer`. Returns the number of frames found.
* `int backtrace_unwind_frame(backtrace_t *buffer, int size, ... registers ...)`
  Unwinds a call stack starting from a specific processor context (e.g., from an exception frame). Register arguments are architecture-specific.

### Name Resolution APIs
* `const char *backtrace_function_name(uint32_t pc)`
  Resolves a program counter (PC) address to a function name string.
* `const char *backtrace_name(uint32_t address)`
  Resolves a function entry point address to a function name string (extracting embedded strings).

### Saving APIs (User-Space Only)
* `void backtrace_save(void)`
  Unwinds the current stack and saves it to the kernel TCB via the `DBGC_SAVEBT` syscall.
* `void backtrace_save_frame(... registers ...)`
  Unwinds the stack from a specific context and saves it to the kernel TCB.

## 3. Integration Guide for Other Architectures

Integrating the backtrace system for a new processor architecture requires implementing the unwinder, configuring the toolchain, and hooking into the exception handlers.

### 3.1 Toolchain and Build Configuration (`mk/gcc.mk`)
1. **Unwinding Information**: Configure the compiler to generate unwinding metadata.
   * If using a table-driven unwinder (like ARM EABI), enable `-funwind-tables`.
   * If using frame pointers, ensure `-fno-omit-frame-pointer` is used.
2. **Name Resolution**: Configure the compiler to embed function names.
   * On GCC for ARM, `-mpoke-function-name` embeds the name before the function entry.
   * If the architecture's compiler does not support this, you may need to disable runtime name resolution and rely on offline tools like `addr2line`.

### 3.2 Linker Scripts
Ensure that the unwinding tables generated by the compiler are kept in the final binary and are locatable at runtime.
* Add the unwinding sections (e.g., `.ARM.exidx`/`.ARM.extab` for ARM, or `.eh_frame` for DWARF) to the architecture's linker script (`kern.ld` and `user.ld`/`user-nommu.ld`).
* Export symbols identifying the start and end of these sections (e.g., `__exidx_start`, `__exidx_end`).

### 3.3 Implementing the Unwinder
You must provide an architecture-specific implementation of the unwinding logic in two places:
* **Kernel**: `sys/lib/backtrace.c`
* **User-Space**: `usr/lib/prex/gen/backtrace.c`

The unwinder implementation must provide:
1. `backtrace_unwind()`
2. `backtrace_unwind_frame()`
3. `backtrace_function_name()` / `backtrace_name()`

For frame-pointer architectures (like x86), this involves walking the linked list of frame pointers (`EBP`/`RBP`). For table-driven architectures, it involves parsing the unwind tables and executing the unwind opcodes.

### 3.4 Hooking Exception Handlers
To ensure backtraces are captured during faults (e.g., Segmentation Fault, Data Abort), update the POSIX signal exception handler for the new architecture in `usr/lib/posix/signal/__exception.c`.

```c
// Example hook for a new architecture
#ifdef __newarch__
    backtrace_save_frame(regs[PC], regs[LR], regs[SP], ...);
    count = backtrace_unwind_frame(bt, 16, regs[PC], regs[LR], regs[SP], ...);
#endif
```

### 3.5 Updating ELF Section Types
If the unwinding tables use a processor-specific ELF section type (like `SHT_ARM_EXIDX`), ensure this type is recognized as loadable during binary execution. Guard these architecture-specific checks appropriately in:
* `bsp/boot/common/elf.c`
* `usr/server/exec/exec_elf.c`
